## 核心结论

1. **信息隐藏是实现深层模块最重要的技术**：每个模块封装若干设计决策知识，将其嵌入实现而非接口，从而简化接口、减少认知负荷，并使设计变更仅影响单个模块。
2. **信息泄漏是软件设计最重要的危险信号**：同一设计决策反映在多个模块中会造成跨模块依赖，发现时应优先考虑合并类或提取独立封装类，避免后门泄漏比接口泄漏更难察觉。
3. **时间分解是信息泄漏的常见诱因**：按操作时序划分模块（先读、再处理、最后写）会将同一知识分散到多个类中；应按"执行任务所需知识"而非"操作发生顺序"来组织模块。
4. **类稍大一些往往能改善信息隐藏**：将相关功能的代码汇聚到同一类（如 HTTP 请求的读取+解析合并），可提高接口级别、减少外部可见方法，使类更深。
5. **接口应使常见情况尽可能简单，提供合理默认值**：类应"自动做正确的事"，避免暴露内部数据结构（如用 `getParameter` 而非 `getParams` 返回 Map 引用）。

## 关键数据

- 信息隐藏最早由 David Parnas 在 1972 年论文 "On the Criteria to be Used in Decomposing Systems into Modules" 中提出
- HTTP POST 请求格式：初始行（含方法/URL/协议版本）+ 若干 Header（含 Content-Length 等）+ 空行分隔 + 可选 Body
- URL 编码示例："+"代表空格，"%21"代表"!"，HTTP 解析器应自动解码后返回给调用者

## 疑点 / 待验证

- 信息隐藏与类大小的权衡边界：第 9 章将讨论何时应拆分为多个小类，当前章节仅强调"稍大一些"的优势，尚未给出量化指引。
- 后门泄漏（如文件格式知识同时被读写类持有）在大型代码库中难以系统性检测，是否有自动化工具辅助发现尚无定论。

## 术语表

- **信息隐藏（Information Hiding）**：每个模块封装若干设计决策知识，实现细节不出现于接口，是实现深层模块的核心技术，由 David Parnas 最早提出。
- **信息泄漏（Information Leakage）**：同一设计决策反映在多个模块中，造成跨模块依赖，是软件设计最重要的危险信号。
- **时间分解（Temporal Decomposition）**：按操作时序组织模块结构的设计风格，易导致同一知识分散到多个类，造成信息泄漏。
- **深层模块（Deep Module）**：功能丰富但接口简单的模块，隐藏大量信息；与信息隐藏互为因果。
- **过度暴露（Overexposure）**：常用特性的 API 迫使用户学习不常用的其他特性，增加不必要的认知负荷。
