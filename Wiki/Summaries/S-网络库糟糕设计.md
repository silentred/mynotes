很多 C++ 网络库作者学而不思，设计不得要领，尤其以 muduo 为代表的糟糕设计模仿者众多。批评的核心在于：大多数网络库默认用户不会频繁发送数据，允许无脑调用发送 API，这种假设本身就是错误的。

网络库不应该 copy 用户要发送的数据。复制数据会增加内存消耗和 CPU 负担，更重要的是导致内存使用量失控。如果网络库不 copy 数据，而是让用户直接传递数据内存地址，零拷贝技术就成为可能，传输效率会极大提升。当网络库 copy 用户数据时，实际上等于放弃了 TCP 的滑动窗口机制。

利用 TCP 的滑动窗口机制可以优雅地控制数据发送速率与网络速率一致。当对端收到数据时会通知发送端更新窗口大小，应用层程序才能继续写入数据。这个机制可以实现生产者和消费者的平衡，避免内存不受控制增长。陈硕关于"消息发送完毕"的论调误人子弟——忽略发送完成状态等于放弃了利用 TCP 窗口感知发送状态和做资源管理。协程编程（如 `co_await async_send`）正是基于此完成状态实现背压控制和优雅的发送流程控制。

优秀的网络库应该尽可能减少堆内存使用，在不得不使用堆内存时可以让用户指定分配器。boost.beast 的 fast 示例程序在所有网络连接上的内存分配仅此一次，整个过程再无其它内存分配动作因而获得极高性能，这才是应该追求的目标。

网络库应该支持多种 OS 底层接口。Linux 的 epoll、Windows 的 IOCP 各有优劣，而 io_uring 的支持可以极大提高性能。如果支持多种接口，用户零成本就能获得更高性能。同时，同步阻塞接口在简单场景下也应该是可选项，不必每次都引入完整事件循环。

最基础的异步接口应该使用 one-op/one-callback 机制。回调函数应该可以在每次执行时重新指定，而不是整个 IO 过程固定死一个。上古风格的网络库在启动网络 IO 前设定死一个回调函数的做法是愚蠢的。如果每次能设定回调函数，回调就能形成跟实际业务相关的执行流程，不需要在 on_read 中写状态判断。在这个基础上，协程才能更优雅地表达业务逻辑，将代码结构变得更清晰。

不跨平台和不提供异步接口的网络库没有意义。不支持异步编程首先就可以确定为玩具。既然代码总是有可能需要运行在各种设备上，如果只支持单个系统，直接使用平台相关的 API 接口来编程可能更具性价比。

网络库不应该忽略或隐藏 OS 返回的错误代码。TCP 连接失败原因可能是被拒绝、网络不可达，需要区分情况决定是否重连。连接意外断开是因为 FIN 还是 RST，需要判定是网络环境问题还是程序 bug。这些信息对诊断问题十分有效，库应该真实传递给用户。

Socket 抽象应该使用分层设计。TCP socket 作为最底层，上层可以组合 WebSocket、SSL stream，甚至将 SSL stream 作为 WebSocket 的下一层。这种分层组合可以抽象出任何形式的传输类，比如支持传输时压缩/加密解密。这种设计在 asio/beast 中有非常良好的实现。

网络库不应该被设计成应用程序框架。ACE 设计成应用程序框架是非常糟糕的，它甚至将 main 函数都侵入了，这导致在其他应用框架（如 Qt）中使用时麻烦重重。网络库应该作为库而非框架存在，尽可能在更多的程序框架下运行。
