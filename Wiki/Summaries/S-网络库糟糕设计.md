## 核心内容总结

1. **网络库不应 copy 用户数据**：复制增加内存消耗与 CPU 负担，且使内存控制复杂化；零拷贝设计让用户直接传递内存地址，可极大提高传输效率，同时使 TCP 滑动窗口机制得到充分利用。
2. **TCP 数据收发应充分利用滑动窗口特性**：忽略"消息发送完毕"事件等于放弃利用 TCP 窗口感知发送状态和做资源管理；协程编程（如 `co_await async_send`）正是基于此完成状态实现背压（Backpressure）控制和消费者-生产者平衡。
3. **内部队列内存分配应由用户指定分配器**：优秀网络库应减少堆内存使用，允许用户接入自定义分配器或静态内存池，boost.beast 的 fast 示例全程仅一次堆分配即典型。
4. **应支持多种 OS 底层接口**：如 Linux epoll、io_uring，Windows IOCP，支持多种接口使用户零成本获得更高性能；简单 TCP 程序也可直接用同步阻塞而不必引入完整事件循环。
5. **异步接口应采用 one-op/one-callback 机制**：每次 I/O 操作独立指定回调，支持业务逻辑自然形成执行流程，避免状态机判断，是接入协程接口（如 `co_await async_read`）的基础；上古风格"一次设定回调贯穿全程"实为误读。
6. **必须跨平台并提供异步接口**：不跨平台则直接用 OS API 更具性价比；不支持异步编程则定位为玩具。
7. **不应隐藏 OS 错误代码**：TCP 连接失败原因（拒绝/不可达）、断开原因（FIN/RST）等对诊断问题至关重要，库应真实传递或做合理抽象（如 error_code）。
8. **Socket 抽象应分层设计（Layer）**：TCP socket 为底层，上层可组合 WebSocket、SSL stream、压缩/加密等；asio/beast 的良好分层设计是范例。
9. **网络库不应被设计为应用程序框架**：框架侵入 main 函数（如 ACE）会导致与其他应用框架（MFC/Qt）难以共存，网络库应作为库而非框架存在。
10. **应使用现代 C++ 而非 C with class**：关注 RAII、智能指针、模板等现代特性。

## 关键术语

- **零拷贝（Zero-Copy）**：网络库不复制用户数据，直接持有用户内存地址进行发送，减少内存复制开销。
- **TCP 滑动窗口（Sliding Window）**：TCP 的流量控制机制，接收端通过窗口大小告知发送端可发送数据量，应用层应感知此状态而非盲目写入。
- **背压（Backpressure）**：下游消费者处理速度跟不上上游生产者时产生的反向压力，利用 TCP 发送完成状态可自然实现背压控制。
- **one-op/one-callback**：每次 I/O 操作独立指定回调，与上古风格的"一次设定回调贯穿全程"相对，支持协程扩展。
- **生产者-消费者平衡**：利用 TCP 窗口控制发送速率，使发送端内存不会因对端处理慢而无限增长。

## 疑点 / 待验证

- io_uring 在高并发场景下的实际性能收益与兼容性之间的权衡尚未在文章中展开。
- 零拷贝实现依赖操作系统支持（如 Linux sendfile/mmap），不同平台的具体实现路径差异较大。
