## 核心内容总结

1. Anthropic 发布官方指南，系统梳理五种多智能体协作模式：生成-验证者、调度-子智能体、智能体团队、消息总线、共享状态，各有适用场景与局限。
2. 选择模式的核心原则是"从最简单的能跑通的模式开始，观察瓶颈后再逐步升级"，推荐从**调度-子智能体**起步，协调成本最低。
3. 选型决策树围绕三个核心问题展开：子任务是否需要长期记忆？工作流是否可预测？智能体间是否需要实时共享发现？
4. 生产系统往往混搭使用多种模式，常见组合是调度-子智能体主管工作流，共享状态处理需要高频协作的子任务，消息总线分发事件。
5. 每种模式都有致命缺陷：生成-验证者易陷入死循环；调度者成信息瓶颈；智能体团队面临资源竞争冲突；消息总线难排查静默崩溃；共享状态存在"反应式死循环"风险。

## 关键数据

- **起步推荐**：调度-子智能体（Orchestrator-subagent），协调成本最低
- **决策维度**：任务是否需要长期记忆 × 工作流是否可预测 × 智能体间是否需要实时对答案
- **停止机制必备**：共享状态模式若无收敛阈值或时间预算，系统会无限循环直到 Token 耗尽

## 简述要点

多智能体协作的本质是**在上下文边界（Context boundary）和信息流动方式之间做权衡**。Anthropic 建议围绕"每个智能体需要知道什么"来拆解工作（Context-centric decomposition），而非按任务类型分工。五种模式由简入繁：生成-验证者通过"修改-审核"循环提升质量，适合评估标准明确的上层输出；调度-子智能体让主智能体统筹规划并分发子任务，各子智能体在独立上下文中执行，Claude Code 采用此模式；智能体团队在此基础上加入成员持久性，成员长期存在并积累领域知识，适合大规模代码迁移等长周期任务；消息总线以发布/订阅机制替代中心协调，适合事件驱动的安全运营等动态流水线；共享状态彻底去中心化，所有智能体读写同一块"黑板"，通过收敛条件终止，最适合需要高度协作的综合研究系统，但必须预设停止机制防止反应式死循环。

## 疑点 / 待验证

- 消息总线中基于 LLM 的路由器失灵（幻觉、理解偏差）在生产环境中的真实概率与规避方案
- 共享状态模式的"一票否决"终止条件如何量化（如收敛阈值设多少算合理）

## 术语表

- **生成-验证者（Generator-Verifier）**：生成者产出结果，验证者检查是否符合标准，不通过则返回反馈，循环迭代直至通过或达到上限；适合质量敏感且评估标准明确的任务
- **调度-子智能体（Orchestrator-Subagent）**：主导调度者规划任务并分发给子智能体，子智能体在独立上下文中执行后汇报结果，调度者汇总；Claude Code 采用此模式
- **智能体团队（Agent Teams）**：协调者创建多个长期存在的团队成员，成员从共享任务队列自主接单，跨任务积累上下文知识；与调度模式的核心区别在于成员持久性
- **消息总线（Message Bus）**：智能体通过发布/订阅与主题路由器交互，实现去中心化的事件驱动协调；适合事件类型不确定、系统需持续扩展新智能体的场景
- **共享状态（Shared State）**：所有智能体读写同一持久化存储（黑板），无中心指挥官；通过收敛阈值或时间预算触发停止，消除单点故障，但存在反应式死循环风险
- **上下文边界（Context Boundary）**：每个智能体所能访问和记忆的信息范围，决定了模式的信息流动方式
- **上下文中心拆解（Context-centric Decomposition）**：按"智能体需要知道什么"划分工作边界，而非按任务类型，是选型和设计的核心原则
- **反应式死循环（Reactive Loops）**：共享状态模式中智能体互相响应对方的新发现形成无限循环，消耗大量 Token 而无法收敛
