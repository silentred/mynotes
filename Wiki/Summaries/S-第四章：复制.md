## 核心结论

1. **复制模式的三元权衡**：同步复制（W=N，强一致但高延迟）、异步复制（W=1，低延迟但数据丢失风险）、半同步复制（W>N/2，折中方案）——不同业务场景对一致性和可用性的取舍决定复制策略。
2. **Quorum 机制是复制的核心约束**：写法定人数 W 和读法定人数 R 满足 W+R>N 保证每次读写至少有一个交集节点，是无主复制的理论基础，也是主从复制中判断提交成功的依据。
3. **一致性模型呈强度梯度**：线性一致性（最强，需实时性）> 顺序一致性（保留程序顺序，无实时性）> 因果一致性（仅保留因果关系）> 最终一致性（最弱，事件最终收敛）——CAP 定理在必须容忍网络分区时，一致性和可用性不可兼得。

## 关键数据

- 复制同步模式对比：同步（W=N）→ 异步（W=1）→ 半同步（W>N/2）
- 线性一致性核心要求：不同节点上非并发事件在重排后必须保持实时先后顺序
- 因果一致性实现手段：向量时钟（节点数决定向量长度，但无法动态适应节点变化）或带父子消息 ID 的逻辑时钟
- Fencing Token（栅栏令牌）：主节点切换时单调递增的令牌号，防止旧主节点"假死"后覆盖新主节点写入，解决 Lease 的活性 + Fencing 的安全双重保障
- 脑裂问题根源：检测节点失效时"双向认知不一致"——单向判断节点下线不足以成为安全依据，必须依赖多数派（Quorum）达成共识

## 疑点 / 待验证

- 因果一致性的向量时钟在节点动态加入/退出时的实现复杂度：无法预知节点数量时，如何初始化和扩展向量时钟数组。
- 半同步复制中"足够多的从节点"的具体阈值设定：在 N=3 或 N=5 集群中，W=2 还是 W=3 能同时兼顾延迟和数据安全性。
- Fencing Token 在非共识系统（如 Redis 主从）中的实际部署效果：客户端是否需要感知 token，还是由存储层自动拦截过期写入。

## 术语表

- **数据复制（Data Replication）**：通过在多个节点冗余存储相同数据，提升可靠性、可用性、性能和容错能力。
- **主从复制（Primary-Backup Replication）**：写请求必须经主节点，再同步到从节点；从节点可处理读请求（可能读到过期数据）。
- **同步 / 异步 / 半同步复制**：按主节点应答客户端时机区分的三种复制模式，依次对应 W=N / W=1 / W>N/2。
- **法定人数（Quorum）**：读/写操作必须获得最小参与节点数才能被系统承认，通常取 N/2+1 以上（多数派）。
- **脑裂（Split-Brain）**：网络分区导致两个节点都认为自己是主节点，同时处理写入，导致数据冲突。
- **租约（Lease）**：检测节点发放给被检测节点的带时限许可证，解决单向心跳可能产生的脑裂问题。
- **Fencing Token（栅栏令牌）**：主节点切换时单调递增的版本号，防止过期主节点覆盖合法数据。
- **顺序一致性（Sequential Consistency）**：所有操作按单一全局顺序执行，保留每个节点程序顺序，无回退。
- **线性一致性（Linearizability）**：顺序一致性 + 实时性（real-time），非并发操作的先后顺序在全局排序中必须得到保持。
- **因果一致性（Causal Consistency）**：仅保证有因果关系的事件顺序，无因果关系的并发事件可任意排序。
